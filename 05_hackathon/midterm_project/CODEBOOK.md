# Codebook: CityPulse Congestion Hub

This document defines the schema and variables used across the synthetic database for the Midterm DL Challenge.

## Database: Supabase PostgreSQL

### Table 1: `locations`
Stores the metadata for the different city intersections and zones monitored by the application.

- **`id`** (UUID, Primary Key): Unique identifier for the location, auto-generated by PostgreSQL.
- **`created_at`** (Timestamptz): Timestamp of when the location was added to the database.
- **`name`** (Text): The human-readable name of the specific monitoring point (e.g., "Downtown Crossing").
- **`zone`** (Text): The broader geographical area the point belongs to (e.g., "Central", "North").

### Table 2: `congestion_readings`
Stores the hourly time-series telemetric data for traffic severity across all monitored locations.

- **`id`** (UUID, Primary Key): Unique identifier for this specific reading.
- **`created_at`** (Timestamptz): Timestamp of when the record was inserted into the database.
- **`location_id`** (UUID, Foreign Key): Links to `locations.id`. Identifies which intersection this reading maps to.
- **`timestamp`** (Timestamptz): The exact historical date and time that the traffic reading represents.
- **`severity_level`** (Integer): A categorical numerical rating of traffic congestion at that timestamp.
  - `1` = Clear / Free-flowing traffic.
  - `2` = Normal traffic.
  - `3` = Moderate congestion / Slowing down.
  - `4` = Heavy congestion / Significant delays.
  - `5` = Gridlock / Stop-and-go.

## Synthetic Data Generation Methodology
The `seed_data.py` script populates the `congestion_readings` table by simulating 365 days of historical data for each location. It mimics realistic organic traffic by establishing a baseline severity depending on the time of day (e.g., high during rush hours, low overnight), and then applies a randomized jitter (`-1` to `+1`) to simulate organic variance while maintaining structural bounds (`1` to `5`).
